# パスキー認証設計

## 方針

このプロジェクトでは、パスワードを使用しません。

認証は Passkey / WebAuthn を前提とします。メールアドレスはログインIDとして使用し、本人確認の秘密情報としては扱いません。

## パスワードを使わない理由

- 高齢者にとってパスワードの作成、記憶、入力、再設定が負担になりやすい
- 使い回しやメモ書きによるリスクがある
- フィッシングや偽ログイン画面への対策として、パスキーを優先する

## メールアドレスをログインIDとして使う理由

- 管理者が参加者を確認しやすい
- 承認や復旧時の連絡先として使える
- パスキー登録前の仮登録情報として扱いやすい

メールアドレスは個人情報です。一般メンバー画面には表示しない方針です。

## 仮登録から承認までの流れ

1. 利用者が仮登録画面を開く
2. 氏名、メールアドレス、卒業時のクラス名を入力する
3. システムが承認待ちユーザーとして保存する
4. 管理者が承認待ち一覧で内容を確認する
5. 管理者が承認または否認する
6. 承認された利用者はパスキー登録へ進む

## 承認後のパスキー登録

1. 承認済みユーザーがメールアドレスを入力する
2. システムがパスキー登録可能な状態か確認する
3. ブラウザの WebAuthn API を使ってパスキーを登録する
4. 公開鍵と資格情報IDを保存する
5. 登録完了後、ユーザー状態を有効にする

## パスキーログイン

1. ユーザーがメールアドレスを入力する
2. システムが登録済みパスキーの有無を確認する
3. ブラウザの本人確認操作を開始する
4. WebAuthn の署名検証に成功した場合にログインする
5. 最終ログイン日時を更新する

## 端末変更・紛失時の復旧

端末変更や紛失によりログインできない場合は、管理者に連絡し、管理者が再登録を許可します。

復旧時もパスワードは発行しません。

## 管理者による再登録許可

- 管理者が対象ユーザーを確認する
- 既存パスキーを必要に応じて無効化する
- ユーザー状態を再登録許可中にする
- ユーザーが新しい端末でパスキーを登録する
- 登録後、通常の有効状態に戻す

## 1ユーザー複数パスキーを想定する理由

- スマートフォンとPCなど複数端末を使う可能性がある
- 端末変更時の復旧負担を減らせる
- 予備端末を登録できると、管理者対応が少なくなる

## 高齢者向け説明文案

パスキーは、パスワードの代わりに、お使いのスマートフォンやパソコンの本人確認機能でログインする仕組みです。

ログイン時には、画面の案内に従って、指紋認証、顔認証、端末のロック解除などを行います。

パスワードを覚えたり入力したりする必要はありません。

端末を変えた場合やログインできなくなった場合は、管理者に連絡してください。
