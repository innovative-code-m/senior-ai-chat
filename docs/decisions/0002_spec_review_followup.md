# 0002 仕様レビュー指摘への対応方針

## 決定日

2026-05-14

## 背景

`docs/reviews/0001_spec_review_2026-05-13.md` で、Phase 2 以降に進む前の仕様不整合と設計ギャップが指摘されました。

特に、ユーザー状態、初期管理者作成、承認後の案内、SignalR接続の認証、反応ボタン、チャット履歴取得の方針を明確にする必要がありました。

## 決定

- Phase 1 は完了扱いとし、現在は Phase 2 着手前の仕様整理・設計更新フェーズとする
- `Users.Status` は `PendingApproval`、`PasskeyRegistrationPending`、`Active`、`Suspended`、`PasskeyResetAllowed`、`Rejected` とする
- `Approved` という単独の Status 値は使わない
- 初期管理者は、管理用DBへの一回限りの手動投入で作成する
- MVPでは承認後の自動メール送信を必須にしない
- WebAuthn後のセッションは HttpOnly Cookie セッションを前提とする
- SignalR Hub は API と同じ Cookie セッションで認証する
- 反応ボタンはトグル操作とし、同じ利用者・同じ投稿・同じ反応種別の重複登録を禁止する
- チャット履歴は最新30件から表示し、「以前の投稿を読む」で30件ずつ追加取得する
- 投稿カテゴリと反応種別は、MVPではマスターテーブル化せず固定文字列で保存する

## 理由

- Phase 2 の管理者承認に入る前に、状態遷移と初期管理者の前提を固定する必要がある
- 自動メール送信をMVP必須にすると、送信基盤、ログ、誤送信対応、個人情報管理の検討が増える
- 小規模 closed グループのMVPでは、管理者の個別連絡と画面上の状態確認で承認後案内を扱える
- Cookie セッションに統一すると、API と SignalR の認証方式を単純にできる
- 反応の重複禁止とトグル操作により、高齢者が誤操作した場合でも戻しやすくなる
- 履歴取得を段階的にすると、初期表示を軽く保ち、操作も分かりやすい

## 今後の確認事項

- 初期管理者を手動投入する具体的なSQLと確認手順
- さくらインターネット環境での Cookie、CORS、HTTPS 設定
- 自動メール通知を将来導入するかどうか
- 投稿ログの保存期間
