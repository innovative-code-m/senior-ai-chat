# 0003 Phase 2 ユーザー仮登録と管理者承認

## 決定日

2026-05-14

## 背景

Phase 2 では、参加希望者の仮登録と管理者による承認・否認の最小フローを実装します。

一方で、Passkey / WebAuthn による本認証、初期管理者の実運用作成、データベース接続はまだ後続 Phase の対象です。無認証の管理者機能を本番利用できる形で残すことはできません。

## 決定

- Phase 2 のデータ保持は、バックエンドプロセス内のインメモリストアに限定する
- データベース作成、接続文字列、マイグレーション、固定シードデータは追加しない
- アプリ再起動で登録データが失われることを Phase 2 の制限として扱う
- 初期管理者レコードは Phase 2 では作成しない
- 管理者 API と管理者画面は `Development` 環境限定のローカル検証機能とする
- 本番環境では無認証の管理者 API を起動しない
- 仮登録時は `PendingApproval`、承認時は `PasskeyRegistrationPending`、否認時は `Rejected` に遷移する
- `Approved` という Status 値は使わない
- メールアドレスは前後空白を除去し、小文字化した値で重複判定する
- 既存メールアドレスでの再申請は、状態にかかわらず Phase 2 では許可しない
- 承認後の自動メール送信は行わず、管理者の個別連絡と状態確認画面で扱う

## 理由

- Phase 2 の目的は、状態遷移と画面・API の最小フローを確認することにある
- データベースをこの時点で追加すると、接続情報、バックアップ、初期管理者投入、移行手順の検討が必要になる
- Passkey / WebAuthn の本認証前に、本番利用可能な無認証管理画面を作ることはできない
- closed グループの参加承認では、同じメールアドレスの再申請を自動で許可するより、管理者が個別に確認するほうが安全である

## Phase 2 で行わないこと

- Passkey / WebAuthn の登録、認証、検証
- パスキーログイン
- チャット閲覧、投稿、反応
- 管理者のお知らせ投稿
- データベース接続、接続文字列、マイグレーション
- 初期管理者の本番投入手順
- 自動メール送信

## 後続 Phase への持ち越し

- 初期管理者を手動投入する具体的な SQL と確認手順
- 管理者ログインと管理者セッション
- パスキー登録完了時の `Active` への遷移
- 否認済み、停止済みユーザーの再申請を許可する場合の運用
- データベース導入後のユニーク制約と管理操作ログ
