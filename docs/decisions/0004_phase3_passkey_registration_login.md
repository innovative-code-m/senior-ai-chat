# 0004 Phase 3 パスキー登録とログイン

## 決定日

2026-05-14

## 背景

Phase 3 では、管理者に承認された利用者がパスキーを登録し、そのパスキーでログインできる最小フローを実装します。

Phase 2 まではインメモリの仮登録・承認状態だけを扱っており、WebAuthn、本番向け管理者認証、データベース接続は未実装でした。

## 決定

- WebAuthn 実装ライブラリは `Fido2.AspNet` 4.0.1 を採用する
- Phase 3 ではデータベースを導入せず、既存のインメモリストアを拡張する
- パスキー情報は `UserPasskeys` 相当のインメモリモデルに保存する
- 秘密鍵そのものは保存せず、資格情報 ID、公開鍵、署名カウンタ、user handle を保存する
- WebAuthn チャレンジは用途、ユーザー、発行時刻、有効期限、発行オプション JSON とともにインメモリで保持する
- チャレンジの有効期限は5分とし、完了 API で一回限り利用として消費する
- ローカル開発の RP ID は `localhost`、許可 Origin は `http://localhost:5086`、`http://localhost:5173`、`http://127.0.0.1:5173` とする
- WebAuthn 検証成功後は HttpOnly Cookie セッションを発行する
- Cookie 名は `sac_session`、有効期限は8時間、`SameSite=Lax`、HTTPS では `Secure=true` とする
- フロントエンドから別オリジンのバックエンドへアクセスする場合は `credentials: 'include'` を使う
- 管理者 API と管理者画面は Phase 2 と同じく `Development` 環境限定に留める
- 本番向け管理者認証、初期管理者作成手順、MySQL 永続化は Phase 3 では実装しない

## 理由

- WebAuthn の署名・attestation・assertion 検証を手書きせず、検証済みライブラリに任せる必要がある
- `Fido2.AspNet` は ASP.NET Identity に依存しない構成でも使えるため、現在の Minimal API と React の構成に合わせやすい
- Phase 3 の目的はローカルでパスキー登録・ログインの流れを確認することであり、MySQL、マイグレーション、初期管理者投入、バックアップ方針まで同時に広げると作業範囲が大きくなる
- 管理者 API を本番向けに開くには、管理者本人のパスキー登録、Role 判定、監査ログ、初期投入手順が必要になるため、Phase 3 では無認証管理機能を Development 限定に閉じたままにする
- HttpOnly Cookie セッションにすると、将来の API と SignalR Hub の認証方式を統一しやすい

## Phase 3 で行わないこと

- パスワード認証、仮パスワード、共通パスワード
- チャット閲覧、投稿、反応、メンバー一覧、お知らせ、投稿削除
- MySQL 接続、接続文字列、マイグレーション
- 初期管理者の固定シードデータ作成
- 本番向け管理者認証と管理者セッション保護
- 自動メール送信
- 画像投稿、ファイル添付、個別DM、既読表示、スタンプ、プッシュ通知、AI自動要約、外部SNS連携

## 制限

- アプリ再起動で登録データ、承認状態、パスキー情報、チャレンジ、セッションが失われる
- ブラウザやOS側に残ったパスキーは、サーバー側の資格情報が失われるとログインに使えない
- Phase 3 のインメモリ保存は本番運用可能な永続化方式ではない
- ローカル WebAuthn 検証はブラウザ、OS、Origin、localhost 条件に依存する

## 後続 Phase への持ち越し

- MySQL 8.4 の `Users`、`UserPasskeys`、セッションまたは認証関連テーブル設計とマイグレーション
- 初期管理者の環境別手動投入手順
- `Role = Admin` かつ `Status = Active` の管理者セッションによる管理者 API 保護
- 管理者によるパスキー再登録許可 UI
- パスキーの複数登録、名称管理、無効化
- 本番ドメイン、HTTPS、Cookie、CORS、ログ保存方針の環境別設定
