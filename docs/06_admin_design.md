# 管理者機能設計

## 管理者の役割

管理者は、参加者が安心して利用できるように、参加者承認、投稿管理、認証復旧、お知らせ投稿を行います。

管理者は個人情報を扱うため、最小限の人員で運用する想定です。

## 初期管理者の作成

最初の管理者は、アプリ内の一般仮登録フローでは作成しません。

初回セットアップ時に、管理用DBへ一回限りの手動投入で初期管理者を作成します。

- `Role = Admin` とする
- `Status = PasskeyRegistrationPending` とする
- 実在する管理者の氏名、メールアドレス、クラス名をリポジトリに含めない
- 初期管理者がパスキー登録を完了した後、参加者承認を開始する
- CLIや固定シードデータによる初期管理者作成はMVPでは採用しない

## 承認待ちユーザー一覧

仮登録されたユーザーを一覧表示します。

表示候補:

- 氏名
- メールアドレス
- 卒業時のクラス名
- 登録日時
- 現在の状態

これらは個人情報を含むため、管理者画面以外には表示しない方針です。

## ユーザー承認

管理者が参加資格を確認し、利用を許可します。

承認後、ユーザー状態は `PasskeyRegistrationPending` となり、パスキー登録へ進める状態になります。

MVPでは承認後の自動メール送信を必須にしません。管理者が必要に応じて個別連絡し、利用者はパスキー登録画面で状態確認を行います。

## ユーザー否認

参加資格を確認できない場合、または登録内容に問題がある場合は否認します。

否認理由を保存するかどうかは未決事項です。

## ユーザー停止

不適切な利用や本人確認上の問題がある場合、管理者はユーザーを停止できます。

停止中ユーザーはチャット閲覧、投稿、反応を行えない方針です。

## 投稿削除

管理者は不適切な投稿や個人情報を含む投稿を削除済み状態にできます。

本文を非表示にし、削除日時と削除者を記録する方針です。

## お知らせ投稿

管理者は、利用ルール、AI利用上の注意、メンテナンス案内などを投稿できます。

お知らせはチャットとは別に確認できるようにする想定です。

## パスキー再登録許可

端末変更や紛失によりログインできないユーザーに対し、管理者がパスキー再登録を許可します。

再登録時は既存パスキーの無効化を検討します。

## 管理操作ログの将来検討

MVP後の拡張として、以下の管理操作ログを検討します。

- ユーザー承認
- ユーザー否認
- ユーザー停止
- 投稿削除
- お知らせ投稿
- パスキー再登録許可

管理操作ログは、トラブル時の確認と運用品質向上のために有効です。
