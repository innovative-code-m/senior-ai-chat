# 仕様レビュー: AGENTS.md / README.md / docs/

- レビュー日: 2026-05-13
- 対象: `AGENTS.md`、`README.md`、`docs/` 配下の全設計文書

## 全体評価

ドキュメントとして整理されており、MVP範囲の一貫性や個人情報方針は丁寧に書かれている。
以下に、重大な不整合・設計上のギャップ・小さな整合性の問題を分類して記載する。

---

## 重大な不整合

### 1. AGENTS.md のフェーズ記載が古い

- `AGENTS.md:13` に「現在は初期構想・設計フェーズです」とある
- `README.md:44` には「現在は Phase 1 です」とある
- `docs/07_mvp_development_plan.md` には Phase 1 の検証記録（2026-05-14完了）が書かれている

Phase 1 は完了済みであるにもかかわらず、AGENTS.md が Phase 0 相当の記述のままになっている。

### 2. Users.Status の値とユーザー状態定義のマッピングが不明確

`docs/01_requirements.md` の自然言語による状態定義と、`docs/03_data_model.md` の Status 列の値が対応していない。

| 要件定義の状態 | データモデルの Status 候補 | 問題 |
| --- | --- | --- |
| 仮登録 | Pending? | 「仮登録」と「承認待ち」が別状態なのに同じ Pending で表現される可能性がある |
| 承認待ち | Pending? | 同上 |
| パスキー未登録 | Approved? | 承認済みだがパスキー未登録という状態が明示されていない |
| 有効 | Active | 対応している |
| 停止 | Suspended | 対応している |
| 再登録許可中 | PasskeyResetAllowed | 対応している |

「仮登録」と「承認待ち」を区別するか、「パスキー未登録」を `Approved` で表現するかを明示的に決定し、DB の Status 値として列挙する必要がある。

---

## 設計上のギャップ

### 3. 管理者の初期作成方法が未決のまま Phase 2 に突入できない

`docs/01_requirements.md` の未決事項に「管理者の初期作成方法」が記載されている。
Phase 2 で「管理者承認」機能を実装するには、最初の管理者がどのように作られるかを先に決定する必要がある。
シードデータ投入、初回設定 CLI、または手動での DB 直接挿入などの選択肢が考えられる。

### 4. パスキー登録の通知手段が未定義

`docs/04_authentication_design.md` の「仮登録から承認までの流れ」に「承認された利用者はパスキー登録へ進む」とあるが、承認をユーザーに伝える手段が書かれていない。
メール通知が必要であれば、現在の技術構成（`README.md`）にメール送信の仕組みを追加する必要がある。

### 5. SignalR 接続の認証方式が未定義

`docs/05_realtime_chat_design.md` では SignalR を利用することが書かれているが、接続時の認証伝搬方式（JWT か Cookie セッションか）に触れていない。
パスキー認証後のセッション管理方式が未定義なため、SignalR の設計が決まらない。

### 6. Reactions の取り消し可否が未定義

`docs/03_data_model.md` の Reactions テーブルに削除フラグがなく、`docs/01_requirements.md` の反応ボタンの説明にもトグル・取り消しへの言及がない。
反応を取り消せるかどうかを要件として明示する必要がある。

### 7. Reactions テーブルにユニーク制約の言及がない

同一ユーザーが同じ投稿に同じ反応を複数回登録できてしまう。
`(MessageId, UserId, ReactionType)` のユニーク制約が必要かどうかをデータモデルに明記すべきである。

### 8. チャット履歴のページネーション未定義

`docs/05_realtime_chat_design.md` に「画面再読み込み時は保存済みメッセージを取得する」とあるが、取得件数の上限やページネーション方式の記載がない。
高齢者向けに操作を簡単にする観点からも、履歴の表示方針を決める必要がある。

---

## 小さな整合性の問題

### 9. Messages.UpdatedAt は現状不要

`docs/03_data_model.md` の `Messages.UpdatedAt` に「編集機能を持つ場合に利用」とあるが、投稿編集はMVP外であり要件に記載がない。
列を持つことの意図（将来拡張のための予約か、削除したほうがよいか）を明記すべきである。

### 10. Announcements.PublishedAt も同様

`docs/03_data_model.md` の `Announcements.PublishedAt` に「未公開管理を行う場合に利用」とあるが、下書き機能は要件に書かれていない。
同上の理由で意図を明記すべきである。

### 11. カテゴリ・反応種別の正式な値定義がない

`docs/03_data_model.md` の `Messages.Category` と `Reactions.ReactionType` がどのような型で保存されるか（文字列固定値・enum・マスターテーブル）の記述がない。
実装時に型の選択でぶれが生じる可能性がある。
