# Phase 2 実行プロンプト: ユーザー仮登録と管理者承認

あなたは `senior-ai-chat` リポジトリの Phase 2 を実行する開発エージェントです。

このプロンプトの目的は、`docs/07_mvp_development_plan.md` の Phase 2「ユーザー仮登録と管理者承認」を、既存ドキュメントの方針に沿って実行することです。

Phase 2 では、参加希望者が仮登録し、管理者が承認または否認できる最小の流れを作ってください。ただし、Passkey / WebAuthn の本実装、パスキーログイン、チャット投稿は Phase 2 の範囲外です。

## 2. Phase 2 の目的

Phase 2 の目的は、closed グループに参加したい利用者を仮登録し、管理者が参加可否を判断できるようにすることです。

完了時点で満たすこと:

- 利用者が氏名、メールアドレス、卒業時のクラス名を入力して仮登録できる
- 仮登録された利用者は `PendingApproval` として扱われる
- 管理者が承認待ち利用者を確認できる
- 管理者が承認すると、利用者は `PasskeyRegistrationPending` になる
- 管理者が否認すると、利用者は `Rejected` になる
- 承認後の自動メール送信は行わない
- Passkey / WebAuthn の本実装は行わない
- 個人情報、実在の同期生情報、秘密情報をリポジトリに含めない

## 4. 重要な制約

### 4.1 プロジェクト方針

- 高齢者が安心して使える closed グループチャットの試作であることを前提にする
- 実在する同期生の氏名、メールアドレス、クラス名、旧姓、居住地などを一切入れない
- サンプル値やテスト値は、実在人物に見えない値にする
- メールアドレスの例は `example.invalid` など実配送されないドメインを使う
- 日本語ドキュメントを基本とする
- コードコメントを書く場合は日本語を優先する

### 4.2 認証方針

- パスワード認証を追加しない
- 管理者用の仮パスワード、共通パスワード、秘密の固定文字列を追加しない
- ログイン画面やサンプル UI にパスワード入力欄を作らない
- 認証は将来の Passkey / WebAuthn 実装を前提にする
- Phase 2 では Passkey / WebAuthn の登録、認証、検証処理を実装しない
- 承認済みユーザーは `PasskeyRegistrationPending` で止め、Phase 3 でパスキー登録へ進める

### 4.3 管理者操作の扱い

Phase 2 は管理者承認を扱いますが、Passkey / WebAuthn による本認証は Phase 3 です。

そのため、管理者機能は本番公開できる状態として扱わないでください。実装前に、次のいずれかの扱いを文書化し、選択した方針に沿って実装してください。

- 管理者 API と管理者画面を `Development` 環境限定のローカル検証機能として実装し、本番環境では起動しない
- 管理者承認のアプリケーションサービスと API の土台だけを作り、実際の保護された管理者画面は Phase 3 以降に回す

どちらを選んでも、無認証の管理者機能を本番利用可能な形で残してはいけません。

### 4.4 MVP範囲

Phase 2 で扱う機能:

- 仮登録
- 承認待ち状態の表示または状態確認
- 管理者による承認
- 管理者による否認
- ユーザー状態管理の最小実装

Phase 2 で扱わない機能:

- Passkey / WebAuthn の本実装
- パスキーログイン
- チャット閲覧
- テキスト投稿
- 投稿カテゴリ
- 反応ボタン
- メンバー一覧
- 管理者による投稿削除
- 管理者からのお知らせ
- 画像投稿
- ファイル添付
- 個別DM
- 既読表示
- スタンプ
- プッシュ通知
- AI自動要約
- 外部SNS連携
- 自動メール送信

### 5.3 バックエンド実装

`src/backend/SeniorAiChat.Api/` に、Phase 2 の最小バックエンド機能を追加してください。

推奨する構成:

- `User` 相当のモデル
- `UserStatus` 相当の列挙または固定値
- `UserRole` 相当の列挙または固定値
- 仮登録用 API
- 状態確認用 API
- 管理者の承認待ち一覧 API
- 管理者の承認 API
- 管理者の否認 API
- ユーザー状態遷移を集約するサービス
- 入力検証と重複チェック

API の例:

- `POST /api/registrations`
- `GET /api/registrations/status?email=...`
- `GET /api/admin/users/pending`
- `POST /api/admin/users/{id}/approve`
- `POST /api/admin/users/{id}/reject`

実際の URL は既存コードとの整合を優先して調整して構いません。ただし、README または関連 docs に記録してください。

状態遷移の基本:

| 操作 | 変更前 | 変更後 |
| --- | --- | --- |
| 仮登録 | なし | `PendingApproval` |
| 承認 | `PendingApproval` | `PasskeyRegistrationPending` |
| 否認 | `PendingApproval` | `Rejected` |

注意:

- `Approved` という Status 値は使わない
- `Active` への遷移は Phase 3 のパスキー登録完了時に扱う
- `Suspended`、`PasskeyResetAllowed` は Phase 2 では将来用の値として残し、操作実装は急がない
- メールアドレスはログイン ID 兼連絡先だが、秘密情報として扱わない
- メールアドレスは大小文字差を吸収できるよう正規化を検討する
- 個人情報をサーバーログへ不用意に出さない

### 5.4 フロントエンド実装

`src/frontend/` に、Phase 2 の最小 UI を追加してください。

推奨する画面または表示状態:

- トップまたは入口画面
- 仮登録フォーム
- 仮登録完了、承認待ち案内
- メールアドレスによる状態確認
- 管理者承認画面

UI 方針:

- 高齢者向けに文字サイズ、余白、ボタンサイズ、説明の分かりやすさを優先する
- 入力項目を増やしすぎない
- エラー文は具体的な次の行動が分かる日本語にする
- パスワード欄を作らない
- 承認後は「パスキー登録は次の Phase で扱う」状態に留める
- 管理者画面が開発用限定の場合は、画面上でも本番運用不可であることを管理者向けに分かる形にする

仮登録フォームの項目:

- 氏名
- メールアドレス
- 卒業時のクラス名

一般メンバー向けの画面には、他者のメールアドレスを表示しないでください。
